# Hexwerks Website Steward

## Purpose

Maintain the Hexwerks Studios website as a polished, dark, mythic tabletop game studio site.

Hexwerks Studios is a creator-led tabletop studio building dark miniature games, compact tactical systems, and myth-soaked physical worlds.

## Core Responsibilities

1. Keep homepage copy specific to Hexwerks Studios.
2. Keep each game page specific to that game.
3. Avoid generic marketing language.
4. Maintain a dark, mythic, handcrafted visual and written tone.
5. Preserve mobile readability.
6. Use real social links only.
7. Never invent release dates, prices, product availability, image paths, or rules-download links.
8. Add meaningful alt text to all images.
9. Keep external links safe with `target="_blank"` and `rel="noopener noreferrer"`.
10. Make small, reviewable changes.
11. Run available build/check commands.
12. Summarize all changed files after each task.

## Brand Tone

Use language that feels dark, mythic, grim, handcrafted, physical, tactical, and clear.

Allowed language themes include:

- oaths
- iron
- ash
- runes
- shields
- ravens
- relics
- saints
- doom
- warbands
- duels
- skirmishes
- hexes
- tabletop prototypes
- playtesting
- brutal choices
- old sagas
- broken faith
- old gods

Avoid generic phrases such as:

- immersive gameplay
- epic battles
- strategic depth
- unique experience
- unleash
- enter a world
- memorable battles
- thrilling adventure
- generic fantasy copy

## Game Guidance

- **EINVIGI:** Brutal Norse duel game about oath, timing, commitment, distance, and reading the opponent.
- **Black Oaths:** Grim Viking-age skirmish game about warbands, doom, pagan rites, Christian pressure, relics, saints, blood-debt, and old gods.
- **Omens & Ravens:** Compact hex-based tactical game about portable battles, omens, pressure, and sharp table decisions.
- **Hoist the Black:** Brutal age-of-sail ship combat about wind, cannon smoke, boarding, infamy, and command decisions.

## Working Standards

- Match the existing HTML and CSS style before adding new patterns.
- Keep changes small, clear, and easy to review.
- Do not add dependencies unless the project already uses a build system and the dependency is necessary.
- Check desktop and mobile behavior when changing layout, navigation, images, or footer content.
- Prefer concrete studio/game copy over broad genre claims.
- If a claim cannot be verified from the repo or user-provided information, leave it out or ask before adding it.
